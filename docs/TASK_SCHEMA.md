# Task Schema

This document defines the JSON structures used by the task pipeline.

## 1) Raw Candidate Schema

Raw candidates are generated by miners and stored in `data/tasks/raw`.

Example file:

- `data/tasks/raw/flask_candidates.json`

Top level candidate fields:

| Field | Type | Description |
|---|---|---|
| `repo_id` | string | Repository id (`flask`, `gin`, `typescript`) |
| `language` | string | Language tag |
| `sha` | string | Commit SHA |
| `message` | string | Commit message |
| `changed_files` | list[string] | Source files modified by commit |
| `cdm` | object | CDM output block |
| `diff_preview` | string | Truncated diff text |

`cdm` object fields:

| Field | Type | Description |
|---|---|---|
| `changed_files` | list[string] | Echo of changed files |
| `required_context_files` | list[string] | Context files outside diff |
| `context_distance_hops` | int | Max shortest path hops |
| `irreducibility_score` | float | CDM irreducibility score |
| `rfs` | float | Reasoning forcing score |
| `cfrd` | float | Cross file reasoning depth, when available |
| `constraint_bearing_files` | list[string] | Files with structural constraints |
| `constraint_types_found` | list[string] | Constraint type tags |
| `required_context_details` | list[object] | Per file signal breakdown |

## 2) Validated Task Schema

Validated tasks are curated records in `data/tasks/validated/*.json`.

The current validated files all include these fields:

| Field | Type | Required | Description |
|---|---|---|---|
| `task_id` | string | yes | Unique id like `flask-001` |
| `repo_id` | string | yes | Repo id |
| `repo_url` | string | yes | Upstream URL |
| `commit_sha` | string | yes | Target commit SHA |
| `language` | string | yes | Task language |
| `task_category` | string | yes | Category label |
| `difficulty_level` | string | yes | `easy`, `medium`, `hard`, `expert` |
| `difficulty_score` | float | yes | Numeric difficulty |
| `context_distance_hops` | int | yes | CDM hop depth |
| `irreducibility_score` | float | yes | CDM irreducibility |
| `required_context_files` | list[string] | yes | Required reads for solver |
| `changed_files` | list[string] | yes | Expected fix files |
| `constraint_types` | list[string] | yes | Constraint class labels |
| `prompt` | string | yes | Task prompt passed to eval |
| `human_annotation_notes` | string | yes | Curation notes |
| `assert_design` | object | yes | Retrieval and symbol assertions |
| `constraint_propagation_targets` | list[object] | yes | CPC target patterns |
| `extraction_method` | string | yes | Curation mode |
| `architecture_pattern` | string | yes | Pattern label |
| `timeout` | int | yes | Eval timeout in ms |

## 3) `assert_design` Block

Typical shape:

```json
{
  "retrieval_check": ["src/flask/globals.py"],
  "constraint_symbols": ["LocalProxy", "accessed", "_session", "RequestContext"],
  "cpc_description": "Human readable statement of key architectural constraints"
}
```

## 4) `constraint_propagation_targets` Block

Each target includes:

| Field | Type | Description |
|---|---|---|
| `path` | string | Target id |
| `pattern` | string | Regex pattern |
| `description` | string | Human explanation |

Example:

```json
{
  "path": "accessed_flag",
  "pattern": "\\.accessed\\s*=\\s*True",
  "description": "accessed must be set in the property"
}
```

## 5) Validator Expectations

`extraction/task_validator.py` enforces:

- required field presence and type checks
- allowed category, difficulty, and language values
- minimum prompt length
- non empty `required_context_files` and `changed_files`
- required context is not a subset of changed files
- file existence checks with git object lookup at commit refs

## 6) Recommended Curation Checklist

1. Prompt states symptom and objective clearly.
2. Required context files are truly necessary.
3. Expected changed files are complete.
4. Constraint symbols and CPC targets are specific and testable.
5. Timeout is realistic for task complexity.
